## 一、核心问题：为什么现有方法不够？

现有的视频世界模型（如 Sora、Wan2.2 等）只能接受**粗粒度**的控制信号（文本提示、键盘输入），无法响应用户在 XR 场景中**精细的手部关节运动**和**头部 6DoF 位姿变化**。这带来了两个根本性矛盾：

1. **纯 2D 条件化的深度模糊问题**：如果只用 2D 手部骨骼图作为条件，模型无法区分"手向前伸"和"手在原地但手指张开"——因为它们在 2D 投影上可能看起来一样。
2. **纯 3D 参数的空间对齐问题**：如果只用 3D 关节角度参数作为条件，模型知道手的精确 3D 姿态，但缺乏与图像像素的空间对应关系，难以精确控制手在画面中的位置。

**核心创新就是提出了一种混合 2D-3D 条件化策略，同时解决这两个矛盾。**

## 二、数学框架：基于 Rectified Flow 的视频扩散模型

### 2.1 基础扩散过程

模型建立在 **Rectified Flow**（整流流）框架上。给定一段干净的视频 $V_0$，首先通过 3D VAE 编码器 $\mathcal{E}$ 将其映射到潜空间：

$$z_0 = \mathcal{E}(V_0)$$

前向加噪过程定义为一条从 $z_0$ 到纯噪声 $\epsilon$ 的**直线插值路径**：

$$z_t = (1-t) \cdot z_0 + t \cdot \epsilon, \quad \epsilon \sim \mathcal{N}(0, I), \quad t \in [0, 1]$$

这里 $t=0$ 对应干净数据，$t=1$ 对应纯高斯噪声。与 DDPM 的马尔可夫链不同，Rectified Flow 的路径是**线性的**，这使得采样更高效。

### 2.2 条件流匹配损失

模型需要学习一个**速度场** $v_\Theta(z_t, t)$，它预测在时间步 $t$ 处沿着去噪方向的"速度"。训练目标是**条件流匹配（Conditional Flow Matching, CFM）损失**：

$$\mathcal{L}_{\text{CFM}} = \mathbb{E}_{t, z_0, \epsilon} \left[ \left\| v_\Theta(z_t, t) - u_t(z_0 \mid \epsilon) \right\|_2^2 \right]$$

其中目标速度 $u_t$ 可以从前向过程解析推导。对 $z_t = (1-t)z_0 + t\epsilon$ 关于 $t$ 求导：

$$u_t = \frac{dz_t}{dt} = \epsilon - z_0$$

所以损失函数实际上是：

$$\mathcal{L}_{\text{CFM}} = \mathbb{E}_{t, z_0, \epsilon} \left[ \left\| v_\Theta(z_t, t) - (\epsilon - z_0) \right\|_2^2 \right]$$

> **直觉理解**：模型在学习"从当前噪声状态 $z_t$ 出发，应该以什么速度、什么方向走，才能回到干净数据 $z_0$"。

## 三、核心创新一：混合 2D-3D 手部条件化策略

这是本文**最关键的技术贡献**。DiT 的输入构造如下：

$$x = \text{patchify}\left(\left[z_r, z_c\right]_{\text{channel}}\right) + \mathcal{E}_{\text{conv}}(H) + \mathcal{E}_{\text{cam}}(P)$$

让我逐项拆解：

### 3.1 2D 分支：渲染手部骨骼图 → VAE 编码

将追踪到的手部关节参数**渲染成 2D 骨骼图像序列**（类似 OpenPose 的可视化），然后通过与视频相同的 3D VAE 编码器编码为潜变量 $z_c$：

$$z_c = \mathcal{E}(\text{Render}_{2D}(\text{HandPose}))$$

然后将 $z_c$ 与原始视频潜变量 $z_r$ 在**通道维度**上拼接：

$$\left[z_r, z_c\right]_{\text{channel}} \in \mathbb{R}^{T \times H \times W \times (C_r + C_c)}$$

再通过 patchify 操作转化为 token 序列。

> **作用**：提供**像素级空间对齐**——骨骼图中手指的位置直接对应生成画面中手指应该出现的位置。这解决了"手在画面中哪里"的问题。

### 3.2 3D 分支：手部关节参数 → 轻量级编码

手部用 UmeTrack 模型表示，每只手包含：
- 手腕的 6DoF 变换（3D 平移 + 3D 旋转）
- 20 个手指关节的旋转角度

总共每只手约 **26 个自由度**，两只手共 ~52 维参数向量 $H$。通过一个轻量级的 **1D 卷积编码器** $\mathcal{E}_{\text{conv}}$ 将其编码为与 DiT token 维度匹配的特征：

$$f_H = \mathcal{E}_{\text{conv}}(H) \in \mathbb{R}^{N \times D}$$

其中 $N$ 是 token 数量，$D$ 是 DiT 的隐藏维度。

> **作用**：提供**精确的 3D 深度和遮挡信息**——即使手指在 2D 投影中重叠，3D 参数仍然能区分不同的关节角度。这解决了"手的精确 3D 姿态是什么"的问题。

### 3.3 为什么混合比单独使用任一分支都好？

| 方法 | MPJPE ↓ | MPVPE ↓ | SSIM ↑ |
| --- | --- | --- | --- |
| 纯 2D 骨骼条件化 | 12.83 mm | 10.87 mm | 0.5309 |
| 纯 3D 参数条件化 | 较差（深度模糊） | 较差 | 较低 |
| **混合 2D-3D（本文）** | **12.23 mm** | **9.10 mm** | **0.5574** |

混合方案在手部姿态精度（MPJPE 降低 4.7%，MPVPE 降低 16.2%）和视频质量（SSIM 提高 5.0%）上全面超越纯 2D 方案。

## 四、核心创新二：Token Addition 条件注入机制

本文系统性地对比了四种将条件信号注入 DiT 的方式：

### 4.1 四种候选策略

**（1）Token Concatenation（Token 拼接）**：
$$\text{Input} = [x_{\text{video}}; x_{\text{cond}}] \in \mathbb{R}^{(N+M) \times D}$$
将条件 token 和视频 token 拼接成更长的序列。问题：增加了序列长度，计算量增大。

**（2）AdaLN Modulation（自适应层归一化调制）**：
$$\text{LayerNorm}(x) \rightarrow \gamma(c) \cdot \text{LayerNorm}(x) + \beta(c)$$
用条件信号 $c$ 生成缩放因子 $\gamma$ 和偏移因子 $\beta$ 来调制每一层的归一化。问题：全局调制，缺乏空间精度。

**（3）Cross-Attention（交叉注意力）**：
$$\text{Attn}(Q=x_{\text{video}}, K=x_{\text{cond}}, V=x_{\text{cond}})$$
视频 token 作为 Query，条件 token 作为 Key/Value。问题：引入额外参数和计算。

**（4）Token Addition（Token 加法，本文选择）**：
$$x = x_{\text{video}} + f_H + f_P$$
直接将编码后的条件特征**逐 token 相加**到视频 token 上。

### 4.2 为什么 Token Addition 最优？

Token Addition 的优势在于：
- **零额外序列长度**：不像拼接那样增加计算量
- **空间对齐的逐 token 调制**：每个空间位置的 token 都接收到对应位置的条件信号，比全局 AdaLN 更精细
- **参数高效**：不引入额外的注意力层参数
- 实验表明它在手部姿态精度指标上比其他三种方法高出 **3-12%**

## 五、核心创新三：6DoF 相机位姿编码

头部位姿（即相机位姿）用 **Plücker 坐标嵌入** 编码。对于每个像素 $(u, v)$，Plücker 坐标定义为：

$$\mathbf{p}(u,v) = (\mathbf{d}, \mathbf{o} \times \mathbf{d}) \in \mathbb{R}^6$$

其中 $\mathbf{o}$ 是相机光心位置，$\mathbf{d}$ 是从光心到像素 $(u,v)$ 的射线方向。这产生一个 $H \times W \times 6$ 的特征图，通过编码器 $\mathcal{E}_{\text{cam}}$ 映射到 token 空间：

$$f_P = \mathcal{E}_{\text{cam}}(P) \in \mathbb{R}^{N \times D}$$

> **直觉理解**：Plücker 坐标为每个像素编码了"这个像素对应的 3D 射线是什么"，从而让模型理解相机的位置和朝向变化应该如何影响画面中的透视变换。

## 六、核心创新四：双向教师 → 因果学生蒸馏

### 6.1 为什么需要蒸馏？

双向（bidirectional）视频扩散模型在生成质量上更好（因为可以同时看到过去和未来的上下文），但**无法用于实时交互**——它需要等到整段视频的所有条件信号都收集完才能生成。

### 6.2 蒸馏流程

1. **教师模型**：双向 DiT，一次性生成完整视频片段，质量最优
2. **学生模型**：因果（causal）自回归 DiT，每次只用**最近 3 帧**作为上下文，生成接下来的 **12 帧**

蒸馏过程让学生模型模仿教师模型的输出分布，同时保持因果结构以支持流式生成。最终系统在 H100 GPU 上达到 **11 FPS**，端到端延迟 **1.4 秒**。
